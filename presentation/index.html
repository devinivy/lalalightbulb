What is "serverless"?

Jerry is right: there are servers being used under the hood.

However, they're abstracted away from us from development through deployment:

  • No physical or virtual machine to point at.
  • No provisioning of "instances" or "containers".
  • Nothing to log-in to or run commands on.

The Serverless framework

You need to define two things:

  1. an exported function.
// handler.js
module.exports.log = (event, context, callback) => {
  console.log(event);
  callback(null, {});
};
  1. some static configuration pointing at that function.
# serverless.yaml
functions:
  log:
    handler: handler.log
    events:
      - iot:
          sql: "SELECT * FROM 'mybutton'"

So what's missing?

The Serverless framework is not designed to jive with non-Serverless applications:

  1. Ignores possibility of sharing resources.
  2. Denies all attempts at runtime configuration.
  3. Doesn't integrate kindly with app code.
  4. Discourages higher-level app architecture.

We don't want our business logic to be stuck in Lambda any more than we want it stuck in REST!

A quick example

aws-node-rest-api-mongodb

Ignores "warm boots." Yes, Lambdas stay up for a bit!

It shouldn't be controversial for them to share resources like database connections. This is a well-established and useful technique!

Potentially sucking-up resources: did the author know that the mongodb driver defaults to a connection pool of 5? Now this is up to 5 connections per invocation.

Lots of noisy, incidental code that should be set-up for us before our handler.

Let's put the "server" back in "serverless"

And let's do it for Jerry.

const Hapi = require('hapi');
const Lalalambda = require('lalalambda');

exports.deployment = async () => {

    const server = Hapi.server();

    await server.register(Lalalambda);

    server.lambda({
        id: 'log',
        options: {
            events: [{
                iot: { sql: `SELECT * FROM 'mybutton'` }
            }],
            handler: (event, { server }) => {

                server.log('button-click', event);
            }
        }
    });

    return server;
};

  • Common configuration
  • Common resources and services
  • Server lifecycle and plugin architecture
  • Co-location of lambda handler and its triggers
  • Leverage to move standard logic/config into a framework

Other funnery

Lambda-ize a route

server.route({
    method: 'get',
    path: '/add/{a}/{b}',
    options: {
        plugins: {
            lalalambda: 'addition'
        },
        handler: (request) => {

            const { a, b } = request.params;

            return Number(a) + Number(b);
        }
    }
});

Lambda-ize the whole shebang

await server.register({
    plugin: Lalalambda,
    options: {
        lambdaize: 'my-service'
    }
})

Architecture of lalalambda

  • hapi plugin

    • Makes lambdas declarable similarly to routes
    • Translates between hapi and AWS API Gateway

  • Serverless plugin

    • Ties hapi server lifecycle into Serverless CLI lifecycle
    • Writes custom handler files to disk
    • Contains Serverless handler logic to initialize server and call into hapi-defined lambdas
